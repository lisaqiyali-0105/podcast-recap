01
On Architecture

One agent can’t
do it all.

Where people stumble with OpenClaw is they think they can throw any task at a single agent and get great results. And then they get really frustrated. I don’t think anybody has really unlocked a general-purpose agent that handles everything well. The reason is simple: context overload. It’s the same thing we’ve seen in every LLM chat interface.

Claire Vo — Lenny’s Podcast

The PM Translation

Claire runs nine agents, each scoped to one lane: Polly handles work, Finn handles family logistics, Sam does sales outreach, Howie preps podcast episodes. They don’t cross streams. Her analogy is Slack channels — you wouldn’t put all company communication in #general, so don’t route all tasks to one agent. Context window is finite; specialized agents make better use of it. This is an architecture decision, not a technical one.

In Practice

Map your agent tasks like you’d map Slack channels. What would you never post in #general? That’s a separate agent. If you’re building agent products: the question isn’t “how do we make one agent smarter” — it’s “what lanes of work should never share context?”

02
On Outcomes

Make the user
look like a winner.

The best way to success in B2B SaaS is getting somebody promoted and making them look good. I think about Howie — he just makes me look good every morning when I have a podcast episode to record. He sends me a reminder. It’s like the most friendly reminder: “Hey Claire, remember your meeting with Al. Just in case you forgot, here’s who Al is. Here’s all the stuff he’s going to show.”

Claire Vo — Lenny’s Podcast

The PM Translation

This is the right north star for any agent UX you’re building. Not “complete the task” — make the user feel like a winner after the interaction. Claire’s agents don’t just execute; they hype her up, flag the scheduling conflict before she misses it, draft the brief she didn’t ask for. For any B2B PM building agent features: the moment isn’t “here’s the output” — it’s “here’s the output, and here’s why you look sharp for knowing this.”

In Practice

Rewrite your agent’s success metric as: “Did the user feel more capable after this interaction?” Audit your current agent outputs against that bar. If the output just completes the task but doesn’t give the user a moment of “I look good because of this,” there’s a UX gap.

03
On Management

Your management skills
transfer directly.

“You don’t need the technical skills. You need role scoping or design — how do we talk to customers, how do we talk to each other. The rest of it we can give to Claude Code to figure out.”

Claire Vo

“Managers, this is your moment. You can design your OpenClaw using the organizational skills you’ve developed over your career. I know how to onboard an employee. I know how to give them a good role. I know how to set them up for success.”

Claire Vo

The PM Translation

Claire ran 20+ years of managing humans before managing agents — and she credits that, not engineering, for her success. The waterline model applies: if your agent keeps doing the wrong thing, it’s a scoping or documentation failure, not a dumb AI. The same diagnostic applies to human teams. Define the job, give it the tools it needs, be explicit about what “great” looks like. Then hold it to that bar.

In Practice

Write your agent’s job description like you’d write a JD for a human hire: title, responsibilities, what “great” looks like, what’s explicitly out of scope. Then check if your soul file or system prompt actually reflects that JD. Mismatch = root cause of most agent failures.

04
On Trust

Earn access.
Don’t grant it.

First you get my calendar, and then you can read my email, and then I guess you could draft some emails, and then you can send the emails. And then why don’t you go to all my meetings for me. I’m going on vacation. You don’t onboard your EA by giving them the password to your email account. They have their own email, their own calendar, and you give them access or permission.

Claire Vo — Lenny’s Podcast

The PM Translation

This isn’t just a security posture — it’s an onboarding design principle. Progressive permission flows aren’t friction; they’re trust scaffolding. For product builders: users who grant full agent access on day one are also the ones who abandon it after the first mistake. Design for graduated trust. Let the agent demonstrate competence at low-stakes tasks first. Expand scope after it’s earned it.

In Practice

Audit your agent’s current permission level. What is it doing that you’ve never consciously approved? Design a “trust level 2” that would be meaningfully more useful and requires demonstrated reliability at level 1 first. Map the permission expansion curve before you build the feature.

05
On Infrastructure

API first.
Browser last.

The open web has been so hardened against bots. There are all these bot-identifying mechanisms, all these hard walls. I just think the web is hostile to agents right now. And we’re going to have to rethink what is the interface of the web to be more agent-friendly, because I think we skip ahead a couple of years and the number one user of websites are going to be people’s agents.

Claire Vo — Lenny’s Podcast

The PM Translation

Claire’s hierarchy for any agent task: API → browser → find the problem behind the problem. Design agent integrations around stable API surfaces, not scraping. The DoorDash reframe she offers is the right PM instinct: if the agent can’t order lunch, maybe it reminds you at 10:30 that you actually like eating at home. When a capability is blocked by the environment, ask whether you’re solving the right problem or just the most obvious one.

In Practice

For every flaky agent integration in your product or personal stack, ask: is there an API for this? If yes, switch. If no, ask what problem the user actually has — and whether there’s a framing of that problem that doesn’t require browser use at all.

06
On Delegation

Let your agent
manage you.

I have my agent assign me Linear tickets for things that I need to do — not that it needs to do. Then that’s in my project management system and we all agree on due dates. So let your agent project-manage your tasks as well. And what I feel with the way I have set up my agents is it’s not just a tool doing work for me. It is a team helping me look better to customers, helping me show up better to my family.

Claire Vo — Lenny’s Podcast

The PM Translation

Most people think delegation ends at “have the agent do it.” Claire’s insight is that for tasks requiring human action (make the call, sign the form, make the decision), a text reminder you’ll ignore is not delegation. A ticket in your actual workflow with a real due date is delegation. The direction of project management flips: you’re not managing the agent, the agent is managing you. That’s the unlock.

In Practice

Pick one recurring human-action task your agent currently just reminds you about via message. Have it create a task in your actual PM system (Notion, Linear, wherever you actually work) with a due date instead. Notice whether the completion rate changes.

07
On Onboarding

Just ramble.
Seriously.

The highest bandwidth API for an LLM is just chatting to it. Do it on your phone and do it in voice notes. When that agent asks you for the first time, “who are you, what do you need?” — click the voice thing and say: I’m Claire, I run ChatPRD, I’ve got this complicated life. You can just kind of ramble. And it actually makes sense of all that and gets you a really good outcome.

Claire Vo — Lenny’s Podcast

The PM Translation

The instinct to write a perfect, structured system prompt is the wrong instinct for onboarding. High-bandwidth, messy natural language produces better results than a carefully engineered structured prompt — because it contains context the structured prompt would never think to include. For product design: your onboarding UX shouldn’t force users into structured fields. The open-ended “who are you and what do you need?” approach OpenClaw uses is worth studying closely.

In Practice

The next time you set up or extend an agent, send a 2-minute voice note describing what you actually want instead of writing a system prompt. Then have the agent draft its own instructions from that. Compare the output to what you would have written yourself.
